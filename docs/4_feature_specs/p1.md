## F-P1.1 ŌĆö Exercise Likes

### Context

Students can like exercises they find helpful, allowing other students to discover useful content. Likes are supported for both authenticated users (tracked by `user_id`) and anonymous visitors (tracked by a browser-generated UUID stored in `localStorage`). The like is a toggle ŌĆö clicking again removes the like. `likeCount` on the `exercises` table is updated incrementally (not recounted). The like button appears on exercise list cards and on the exercise detail page.

### Acceptance Criteria

**Backend:**
- Given an authenticated student calls `POST /api/v1/exercises/{id}/like` and has not yet liked the exercise, then a like row is created with `user_id`; `likeCount` increments; response `{ "liked": true, "likeCount": N }`
- Given the same student calls `POST /api/v1/exercises/{id}/like` again, then the like row is deleted; `likeCount` decrements; response `{ "liked": false, "likeCount": N-1 }`
- Given an unauthenticated request with a `browserId` header, when the exercise is liked, then a like row is created with `browser_id` and null `user_id`; deduplication uses `uk_like_browser` unique index
- Given `GET /api/v1/student/exercises/{id}` by an authenticated student who has liked the exercise, then `"liked": true` is included in the response

**Frontend:**
- Given the student views the exercise list, when the page loads, then each card shows a ŌÖź icon with the like count
- Given the student clicks the like button on an un-liked exercise, then the icon fills and the count increments optimistically before the API responds
- Given the API call fails, when the optimistic update is rolled back, then the icon reverts and a subtle error toast is shown
- Given the exercise detail page, then the like button is also present

### Tech Constraints

- `browserId`: client-generated UUID stored in `localStorage`; max 128 chars; sent as `X-Browser-Id` request header
- `exercise_likes` unique constraints enforce deduplication at DB level: `uk_like_user (exercise_id, user_id)` and `uk_like_browser (exercise_id, browser_id)`
- `likeCount` updated via `UPDATE exercises SET like_count = like_count ┬▒ 1` (not a recount query)
- Optimistic UI update before API response; rollback on failure
- Like button accessible without login (anonymous like flow)
- `POST /api/v1/exercises/{id}/like` endpoint accessible to unauthenticated users (no JWT required for anonymous case); authenticated users use their token

### Out of Scope

- Like counts visible to tutors in the management UI (already shown via `likeCount` field in exercise list ŌĆö no additional work)
- Sorting exercises by most-liked (not in this release)
- Dislike / downvote

### API Contract

```
POST /api/v1/exercises/{id}/like
Headers: Authorization: Bearer <token> (optional) | X-Browser-Id: <uuid> (required if unauthenticated)
Response 200: { "liked": true, "likeCount": 13 }

GET /api/v1/student/exercises/{id}
Response 200: (existing endpoint ŌĆö no change except "liked" field now reflects actual state)
{ ..., "likeCount": 8, "liked": true }
```

### File / Class Locations

- `com.platform.exercise.like.LikeController`
- `com.platform.exercise.like.LikeService`
- `com.platform.exercise.domain.ExerciseLike` ŌĆö JPA entity
- `com.platform.exercise.repository.ExerciseLikeRepository`
- Frontend: `src/components/student/LikeButton.jsx` (used in exercise list cards and detail page)

### Dependencies on Other Features

- F-4 (Exercise Management) ŌĆö `exercises` table, `like_count` column
- F-5 (Student Practice) ŌĆö exercise list and detail pages where the button is rendered

## F-P1.2 ŌĆö PythonTutor Integration

### Context

Students working on Python exercises can click "Visualize Execution" to open an embedded PythonTutor visualization in a modal. The current editor code is passed as a URL parameter to PythonTutor's iframe embed. This is a pure frontend integration ŌĆö no backend changes required. The modal is closable and the iframe is destroyed on close to free memory.

### Acceptance Criteria

- Given the student is on a Python exercise page, when they click "Visualize Execution", then a modal opens with an iframe embedding `https://pythontutor.com/iframe-embed.html` pre-populated with the current editor code
- Given the iframe loads, when the student steps through execution, then variable state and call stack are shown step by step without leaving the platform
- Given the code editor is empty, when the button is clicked, then the button is disabled with a tooltip: "Enter some code first"
- Given the modal is closed, then the iframe is destroyed (removed from DOM) to free memory

### Tech Constraints

- PythonTutor URL parameters: `code=<urlencoded>`, `mode=display`, `origin=opt-frontend.appspot.com`
- Full URL pattern: `https://pythontutor.com/iframe-embed.html#code=<urlencoded_code>&mode=display&origin=opt-frontend.appspot.com`
- Modal must be closable via button and `Esc` key
- Iframe is injected into DOM when modal opens; removed from DOM when modal closes (not just hidden)
- No backend changes required ŌĆö pure frontend integration
- CSP: PythonTutor iframe requires adding `https://pythontutor.com` to `frame-src` in Nginx CSP header

### Out of Scope

- PythonTutor integration for Blockly exercises
- Offline or self-hosted PythonTutor instance
- Saving or sharing visualizations

### API Contract

No API changes. Frontend-only feature.

### File / Class Locations

- Frontend: `src/components/student/PythonTutorModal.jsx`
- Nginx CSP update: `src/nginx/nginx.conf` ŌĆö add `frame-src 'self' https://pythontutor.com` to CSP header

### Dependencies on Other Features

- F-5 (Student Practice) ŌĆö specifically `PythonPracticePage.jsx` where the button is added

## F-P1.3 ŌĆö Profile Management

### Context

Any logged-in user can update their display name and change their password from a profile page accessible via the top navigation. Changing the password invalidates all other active sessions (all refresh tokens for that user, except the current session). Display name changes are reflected immediately across the platform without re-login.

### Acceptance Criteria

**Backend:**
- Given `PUT /api/v1/profile` with a new `displayName`, when saved, then `users.display_name` is updated; subsequent API responses reflect the new name immediately
- Given `PUT /api/v1/profile/password` with correct `currentPassword` and valid `newPassword`, when submitted, then password hash updated and all `refresh_tokens` rows for that user **except the current session** are deleted
- Given `PUT /api/v1/profile/password` with incorrect `currentPassword`, then 400 `INVALID_CREDENTIALS`
- Given `newPassword` fewer than 8 characters or missing required character class, then 400 `VALIDATION_ERROR`

**Frontend:**
- Given any logged-in user clicks their name in the top navigation, when the profile page opens, then their current display name and username (read-only) are shown
- Given the user edits display name and saves, when saved, then the name in the top navigation bar updates immediately without a page reload
- Given the user submits the password change form with correct current password, when saved, then a success toast appears; user remains logged in
- Given two separate form sections on the page (display name section and password section), then they are not combined into a single form

### Tech Constraints

- `newPassword` validation: min 8 chars, at least one letter and one digit (server-side authoritative; client-side mirrors for UX)
- Password change: delete all `refresh_tokens` rows for `user_id` WHERE `id != currentRefreshTokenId` (current session preserved)
- Display name change: no caching issue since fetched from DB on each request
- Frontend password form: three fields ŌĆö current password, new password, confirm new password; confirm validated client-side only (not sent to server)
- Username field is display-only (no input element)
- Two separate form sections: `DisplayNameForm` and `PasswordChangeForm`

### Out of Scope

- Username change (username is permanent)
- Profile photo / avatar
- Email address management (platform has no email concept)
- Account deletion

### API Contract

```
GET /api/v1/profile
Response 200: { "id": 1, "username": "alex01", "displayName": "Alex Chen", "role": "STUDENT" }

PUT /api/v1/profile
Request:  { "displayName": "Alex C." }
Response 200: { "id": 1, "username": "alex01", "displayName": "Alex C.", "role": "STUDENT" }
Error: 400 VALIDATION_ERROR

PUT /api/v1/profile/password
Request:  { "currentPassword": "oldPass123", "newPassword": "newPass456" }
Response 200: { "message": "Password updated. Other sessions have been invalidated." }
Errors: 400 INVALID_CREDENTIALS (wrong current password), 400 VALIDATION_ERROR (weak new password)
```

- `com.platform.exercise.profile.ProfileService`
- Frontend: `src/pages/ProfilePage.jsx`, `src/components/profile/DisplayNameForm.jsx`, `src/components/profile/PasswordChangeForm.jsx`
- Top navigation update: `src/components/layout/TopNav.jsx` ŌĆö display name must re-fetch or update from auth context after save
