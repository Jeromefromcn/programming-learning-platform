## F-1 — Infrastructure & DevOps

### Context

The platform runs on a single university server using Docker Compose. Six services must be wired together on a custom bridge network: Nginx (frontend + reverse proxy), Spring Boot API, Python sandbox (nsjail), MySQL, Prometheus, and Grafana. There is no cloud infrastructure — the IT team is not DevOps-specialist and needs a simple `docker compose up` workflow. Database migrations are managed by Flyway. A daily backup job runs `mysqldump` to a host-mounted directory.

### Acceptance Criteria

**Backend skeleton:**
- Given the repository is cloned, when `mvn spring-boot:run` is executed, then the application starts on port 8080 and `GET /actuator/health` returns `{"status":"UP"}`
- Given the project POM, when reviewed, then it declares: Spring Boot Starter Web, Security, Data JPA, Actuator, Validation; JJWT 0.12.6; Flyway; Lombok; Logback Logstash Encoder; H2 (test scope); MySQL Connector; Micrometer Prometheus; Apache Commons CSV; Rhino 1.7.x

**Frontend skeleton:**
- Given `npm run dev` is executed, when the browser opens `http://localhost:5173`, then a login page placeholder renders without console errors
- Given `npm run build`, when executed, then a `dist/` folder is produced that can be served by Nginx

**Docker Compose:**
- Given `docker compose up -d` on a clean machine, when all containers start, then Nginx is reachable on port 80, Prometheus on 9090, Grafana on 3001; Spring Boot and sandbox are internal-only
- Given the MySQL container restarts, when Spring Boot reconnects, then it re-establishes without manual intervention

**Flyway migration:**
- Given a clean MySQL 8.0 database, when Spring Boot starts, then Flyway applies V1 and all 11 tables exist: `users`, `refresh_tokens`, `courses`, `course_students`, `exercises`, `exercise_versions`, `course_exercises`, `submissions`, `categories`, `exercise_likes`, `global_settings`
- Given H2 in-memory test profile, when tests run, then the same migration applies successfully

**Nginx:**
- Given a request to `/api/v1/actuator/health` through Nginx port 80, then it proxies to `api-server:8080`
- Given a request to any non-`/api/*` path, then Nginx serves `index.html` (SPA fallback)
- Given a file upload to `/api/v1/submissions/import` up to 50 MB, then Nginx does not reject it

**Backup:**
- Given the backup container is running, when the scheduled time arrives, then a `.sql.gz` file appears named `backup_YYYY-MM-DD_HH-MM.sql.gz`
- Given 31 days of backup files exist, when the next backup runs, then files older than 30 days are deleted

### Tech Constraints

- Java 17 (Eclipse Temurin), Maven 3.9.x; package root: `com.platform.exercise`; config via `application.yml`
- Lombok annotation processing configured; `@Data` / `@Builder` on entities
- React 18.3.1, Vite 5.x, React Router 6.x; Axios for HTTP (no other HTTP libraries)
- Frontend folder structure: `src/pages/`, `src/components/`, `src/api/`, `src/hooks/`, `src/workers/`
- Docker network name: `exercise-platform-net`; MySQL data persisted to named volume `mysql-data`
- Spring Boot waits for MySQL health check before starting (`condition: service_healthy`)
- No external ports for MySQL (3306), Spring Boot (8080), or sandbox (5000)
- Nginx: `nginx:1.25-alpine`; gzip compression enabled for JS/CSS; proxy headers `X-Real-IP`, `X-Forwarded-For`; `client_max_body_size 50m`
- Flyway migration file: `src/main/resources/db/migration/V1__create_schema.sql`; seed: `INSERT INTO global_settings (setting_key, setting_value) VALUES ('course_filter_enabled', 'false')`
- Python sandbox: `python:3.12-alpine`, nsjail 3.4; Flask on port 5000 (internal only); nsjail: network disabled, filesystem read-only, `/tmp` writable (10 MB), memory 128 MB, PID limit 32
- Backup image: `mysql:8.0`; cron via Alpine `crond`; default schedule: daily 02:00; host dir: `/var/backups/exercise-platform`

### Out of Scope

- Cloud deployment, Kubernetes, load balancing
- External object storage (MinIO/S3) — student files are parsed and discarded
- Multi-server or high-availability setup

## F-2.1 — Login / Logout

### Context

All platform users (STUDENT, TUTOR, SUPER_ADMIN) authenticate with a username and password. The platform uses JWT-based auth: a short-lived access token (15 min) stored in memory, and a long-lived refresh token (7 days) stored as an HttpOnly cookie. Every authenticated request checks the user's status against the DB to support instant account disable without Redis. After login, each role is redirected to their appropriate home page.

### Acceptance Criteria

- Given valid credentials, when `POST /api/v1/auth/login`, then 200 with `accessToken` + user info in body; `refreshToken` in `Set-Cookie` header; user redirected to role-appropriate page within 1 second
- Given incorrect credentials, when the login request is submitted, then 401 `INVALID_CREDENTIALS` (same message regardless of which field is wrong); password field cleared in UI
- Given a disabled account, when login is attempted, then 403 `ACCOUNT_DISABLED`; message shown: "Account disabled — please contact an administrator"
- Given an expired access token while the user is active, when the next API call is made, then token is silently refreshed via refresh cookie without the user seeing a logout
- Given a valid refresh cookie, when `POST /api/v1/auth/refresh`, then 200 with new access token
- Given the user clicks Logout, when confirmed, then `POST /api/v1/auth/logout` is called, session is cleared, user redirected to login page
- Given a STUDENT token, when calling a TUTOR-only endpoint, then 403 `ACCESS_DENIED`
- Given a missing or malformed Authorization header, when calling any protected endpoint, then 401

### Tech Constraints

- JJWT 0.12.6; algorithm: HS256; secret from environment variable
- Access token: 15-minute expiry, contains `userId`, `role`, `jti`; stored in memory only (not `localStorage` — XSS protection)
- Refresh token: 7-day expiry; stored as `SHA-256(token)` hash in `refresh_tokens` DB table; raw token sent as `HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=604800` cookie
- Every authenticated request: DB status check on `users.status` (ADR-3 — no Redis); sub-millisecond on indexed column
- `refresh_tokens` cleanup: scheduled job deletes expired rows daily
- Spring Security: disable CSRF (JWT-based), stateless session; JWT filter applied before `UsernamePasswordAuthenticationFilter`; login endpoint permits all (no JWT filter)
- Rate limit on login: 10 req/min per IP (Bucket4j)
- Role hierarchy via Spring Security `RoleHierarchy` bean: `SUPER_ADMIN > TUTOR > STUDENT`; enforced via `@EnableMethodSecurity` + `@PreAuthorize`
- Axios interceptors: request interceptor attaches `Authorization: Bearer`; response interceptor on 401 attempts one silent refresh, on failure redirects to login
- React Router `<ProtectedRoute>` checks auth and role before rendering
- BCrypt strength factor: 12

### Out of Scope

- SSO / OAuth / SAML
- Password reset via email
- Two-factor authentication
- Student self-registration

### API Contract

```
POST /api/v1/auth/login
Request:  { "username": "student", "password": "student" }
Response 200: { "accessToken": "eyJhbG...", "user": { "id": 1, "username": "student", "displayName": "Alex Chen", "role": "STUDENT" } }
+ Set-Cookie: refreshToken=<token>; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=604800
Errors: 401 INVALID_CREDENTIALS, 403 ACCOUNT_DISABLED

POST /api/v1/auth/refresh
Request: (no body — refresh token read from cookie)
Response 200: { "accessToken": "eyJhbG..." }
Error: 401 TOKEN_EXPIRED

POST /api/v1/auth/logout
Request: (no body)
Response 204: No content. Clears refresh token cookie.
```

## F-2.2 — User Management (SUPER_ADMIN)

### Context

The IT Admin (SUPER_ADMIN) creates and manages all user accounts. There is no self-registration. Tutors manage enrollment, but SUPER_ADMIN controls who can access the platform and in what role. Disabling a user must immediately invalidate all their active sessions.

### Acceptance Criteria

- Given SUPER_ADMIN opens the user management page, when the page loads, then all users are listed with username, display name, role badge, and status
- Given `POST /api/v1/users` with valid data, when username is unique, then 201 with created user; user can log in immediately
- Given `POST /api/v1/users` with a duplicate username, then 409 `USERNAME_TAKEN`
- Given SUPER_ADMIN disables a user, when confirmed, then `users.status = 'DISABLED'` and all rows in `refresh_tokens` for that user are deleted atomically in one transaction; any existing sessions immediately fail the per-request status check
- Given SUPER_ADMIN changes a user's role, when the user next logs in, then they have updated permissions
- Given admin attempts to disable themselves, then blocked with inline error
- Given `GET /api/v1/users`, when called with `role` filter, then only matching users returned (paginated)

### Tech Constraints

- Password set by admin at creation; response must never include `password_hash`
- `PATCH /api/v1/users/{id}/status` to `DISABLED`: delete all `refresh_tokens` rows for that user + set `users.status = 'DISABLED'` in a single `@Transactional` method
- Role change takes effect on next login (new JWT issued at that point)
- `GET /api/v1/users` supports pagination (`page`, `size`) and filter by `role` and `status`
- UI: paginated table; create user modal; role change dropdown; disable/enable toggle with confirmation dialog
- Admin cannot disable their own account (client-side + server-side guard)

### Out of Scope

- Bulk user import (CSV upload of users)
- User profile editing by admin (only role and status)
- Audit log of admin actions

### API Contract

```
GET /api/v1/users?role=STUDENT&status=ACTIVE&page=0&size=20
Response 200: { "content": [{ "id": 1, "username": "alex01", "displayName": "Alex Chen", "role": "STUDENT", "status": "ACTIVE", "createdAt": "..." }], "page": 0, "size": 20, "totalElements": 7, "totalPages": 1 }

POST /api/v1/users
Request:  { "username": "newstudent01", "displayName": "New Student", "password": "initialPass123", "role": "STUDENT" }
Response 201: { "id": 8, "username": "newstudent01", "displayName": "New Student", "role": "STUDENT", "status": "ACTIVE" }
Errors: 409 USERNAME_TAKEN, 400 VALIDATION_ERROR

PATCH /api/v1/users/{id}/role
Request:  { "role": "TUTOR" }
Response 200: { "id": 1, "username": "alex01", "role": "TUTOR", "message": "Role updated. Takes effect on next login." }

PATCH /api/v1/users/{id}/status
Request:  { "status": "DISABLED" }
Response 200: { "id": 1, "username": "alex01", "status": "DISABLED", "message": "User disabled. All active sessions have been invalidated." }
```

## F-3.1 — Category Management

### Context

Categories represent knowledge topics (e.g., "Loops", "Variables", "Conditionals") used to tag exercises. Tutors manage categories directly — no IT involvement needed. Categories appear in the exercise authoring dropdown and in student filter controls. A category cannot be deleted if it has linked exercises.

### Acceptance Criteria

- Given the tutor adds a new category, when submitted, then the category appears in the exercise authoring page dropdown immediately
- Given the tutor adds a category with a name that already exists, then 409 `CATEGORY_DUPLICATE`; message: "This category already exists"
- Given a category has exercises linked to it, when attempting to delete, then 409 `CATEGORY_HAS_EXERCISES`; message: "This category has N exercises — please remove associations first"
- Given `GET /api/v1/categories`, when called, then all categories returned with `exerciseCount` per category

### Tech Constraints

- `categories` table: `UNIQUE INDEX uk_category_name (name)` enforces deduplication at DB level
- `CATEGORY_HAS_EXERCISES` check: query `exercises` table for `category_id = id AND is_deleted = false` before delete
- `GET /api/v1/categories` is accessible to all authenticated roles (needed for exercise authoring dropdowns and student filter)
- POST and DELETE restricted to `TUTOR+` via `@PreAuthorize`

### Out of Scope

- Category hierarchy (no parent/child categories)
- Category reordering
- Category descriptions

### API Contract

```
GET /api/v1/categories
Response 200: [{ "id": 1, "name": "Loops", "exerciseCount": 3 }, { "id": 2, "name": "Variables", "exerciseCount": 2 }]

POST /api/v1/categories

Response 201: { "id": 5, "name": "Recursion", "exerciseCount": 0 }
Error: 409 CATEGORY_DUPLICATE

DELETE /api/v1/categories/{id}
Response 204: No content.
Error: 409 CATEGORY_HAS_EXERCISES
```

## F-3.2 — Course Management

### Context

Courses are the organizational layer between exercises and students. Tutors create courses, link exercises to them, and enroll students. The global course filter (controlled by SUPER_ADMIN) determines whether students can only see exercises from their enrolled courses. Courses support soft-delete — linked exercises, submissions, and grades are retained. Student self-enrollment does not exist; tutors manage enrollment manually.

### Acceptance Criteria

- Given the tutor creates a course, when submitted with name and description, then 201; course appears in course list
- Given the tutor soft-deletes a course, when confirmed, then the course disappears from the list; all linked data (exercises, submissions) is retained
- Given `POST /api/v1/courses/{courseId}/exercises` with exercise IDs, when submitted, then exercises are linked to the course
- Given `DELETE /api/v1/courses/{courseId}/exercises/{exerciseId}`, then the exercise is unlinked; exercise and submissions unaffected
- Given `POST /api/v1/courses/{courseId}/students` with user IDs, when submitted, then students are enrolled
- Given `DELETE /api/v1/courses/{courseId}/students/{userId}`, then the student is unenrolled; historical data retained
- Given `GET /api/v1/courses`, when called, then paginated list returned with `exerciseCount` and `studentCount` per course

### Tech Constraints

- `courses.is_deleted = true` for soft-delete; all list queries filter `is_deleted = false`
- `course_exercises` join table: `(course_id, exercise_id)` unique constraint prevents duplicate links
- `course_students` join table: `(course_id, user_id)` unique constraint prevents duplicate enrollment
- `exerciseCount` and `studentCount` computed via JOIN or subquery (not N+1)
- All course management endpoints restricted to `TUTOR+` via `@PreAuthorize`

### Out of Scope

- Student self-enrollment (invite codes, registration links)
- Course content (videos, reading materials)
- Course archive vs delete distinction

### API Contract

```
GET /api/v1/courses?page=0&size=20

Request:  { "name": "CS301 — Algorithms", "description": "Advanced algorithms course" }
Response 201: Course object

PUT /api/v1/courses/{id}
Request:  { "name": "...", "description": "..." }
Response 200: Updated course object

DELETE /api/v1/courses/{id}
Response 204: No content (soft-delete)

POST /api/v1/courses/{courseId}/exercises

Response 200: { "linked": 3, "message": "Exercises linked to course." }

DELETE /api/v1/courses/{courseId}/exercises/{exerciseId}
Response 204

GET /api/v1/courses/{courseId}/exercises

POST /api/v1/courses/{courseId}/students

Response 200: { "enrolled": 4, "message": "Students enrolled." }
```

## F-4 — Exercise Management

### Context

Tutors author, edit, publish, and version-control exercises. Two types exist: **Blockly** (visual block-based, configured with allowed blocks, initial workspace XML, grading rules) and **Python** (text-based, configured with starter code, test cases, time limit). Every edit creates a new immutable version; tutors can roll back to any previous version. Exercises start as DRAFT and must be explicitly published to become visible to students. Soft-delete retains all associated submissions.

### Acceptance Criteria

**CRUD & Versioning:**
- Given `POST /api/v1/exercises` with valid Blockly or Python config, when submitted, then 201 with `status: "DRAFT"`, `versionNumber: 1`
- Given `PUT /api/v1/exercises/{id}`, when submitted, then a new immutable `exercise_versions` row is created with incremented `versionNumber`; `exercises.current_version_id` updated
- Given `DELETE /api/v1/exercises/{id}`, then `exercises.is_deleted = true`; all GET requests for that ID return 404
- Given `GET /api/v1/exercises/{id}/versions`, then all version rows returned newest-first with `isCurrent` flag on the active version
- Given `POST /api/v1/exercises/{id}/rollback` with a valid `versionId`, then `exercises.current_version_id` updated to that version; no new version row created
- Given a rollback `versionId` that belongs to a different exercise, then 400 `VALIDATION_ERROR`

**Publish / Unpublish:**
- Given `PATCH /api/v1/exercises/{id}/publish` on a DRAFT exercise, then `status = PUBLISHED`; immediately visible to students
- Given `PATCH /api/v1/exercises/{id}/unpublish`, then `status = DRAFT`; immediately disappears from student exercise list
- Given publish on an already-published exercise, then idempotent 200

**Authoring UI:**
- Given the tutor opens "New Exercise" and selects Blockly, when the form renders, then a live Blockly workspace is shown for setting `initialWorkspaceXml` and configuring `allowedBlocks` via a checklist
- Given the tutor configures allowed blocks, when selected, then the Blockly toolbar updates to show only those blocks
- Given the tutor toggles "Show Python Code View", when enabled, then a read-only panel shows the Python equivalent of the current workspace (live update)
- Given the tutor opens "New Exercise" and selects Python, when the form renders, then a Monaco editor is shown with Python syntax highlighting
- Given the tutor clicks "Verify Test Cases", when called, then each test case shows pass/fail result inline (runs against sandbox)
- Given the tutor opens an exercise with multiple versions, when the Versions tab opens, then all versions listed newest-first; rollback confirmation warns: "This will change the exercise students see."

**Validation:**
- Blockly config must have `allowedBlocks` (non-empty array)
- Python config must have at least one test case

### Tech Constraints

- Blockly 12.5.0; `initialWorkspaceXml` captured via `Blockly.Xml.workspaceToDom()` serialised to string
- Monaco Editor 0.50.x loaded via dynamic `import()` (code-split to avoid blocking initial bundle)
- `PUT` creates a new version even if config is identical (no diff-check)
- `exercise_versions` rows are immutable — never updated after creation
- Rollback is a pointer update only on `exercises.current_version_id`; status (DRAFT/PUBLISHED) unchanged after rollback
- Status change reflected in student list within one request (no caching on exercise status)
- Grading rule toggles: Output Match (expected output field), Required Blocks (multi-select), Forbidden Blocks (multi-select), Block Count Limit (number input)
- Python time limit input: integer, 1–30 seconds, default 5
- Exercise list UI: pagination 20 per page; filter dropdowns for type, status, category, difficulty; debounced title search 300 ms
- "Verify Test Cases" calls `POST /api/v1/exercises/verify` (or dedicated preview endpoint) — runs through Python sandbox

### Out of Scope

- Exercise import/export as files (authoring is always done in the UI)
- Exercise duplication / clone
- Student-facing version history
- Rich text in problem descriptions (plain text only)
- Exercise difficulty auto-suggestion

### API Contract

```

Response 200: { "content": [{ "id": 1, "title": "Print Hello World", "type": "BLOCKLY", "difficulty": "EASY", "category": { "id": 2, "name": "Variables" }, "currentVersionNumber": 3, "status": "PUBLISHED", "likeCount": 12, "createdAt": "..." }] }
```

## F-5 — Student Practice

### Context

Students browse published exercises (filtered by course, type, category, difficulty), open an exercise in the appropriate editor (Blockly workspace or Monaco Python editor), run their code locally in-browser using a Web Worker (Blockly → JS execution, Python → Pyodide WASM), and export their answer as a JSON file. ## F-6 — Submission & Grading

### Context

Tutors collect student JSON answer files from OLE and import them into the platform (one or more JSON files, or a single ZIP). The platform validates each file, auto-grades it (Blockly via Rhino JS engine, Python via nsjail sandbox container), and returns a per-file result. Tutors review submissions, optionally apply a manual score and comment (overrides auto-grade), and export grades as CSV to upload back to OLE.

Edge cases: duplicate submissions are flagged and require force-import; version mismatches (student answered an older version) are annotated; malicious ZIP paths are rejected; sandbox unavailability degrades gracefully.

### Acceptance Criteria

**Import Pipeline:**
- Given a valid student JSON file, when imported, then parsed, exercise looked up, auto-graded, and saved; result `status: "IMPORTED"`
- Given a file referencing a deleted or non-existent exercise, then result `status: "FAILED"`, message: "Exercise not found or has been deleted."
- Given a duplicate file (same `studentName` + `exerciseId` + `exportedAt`), then result `status: "DUPLICATE"`; no new DB row created
- Given a ZIP file upload, then extracted; each JSON processed individually; path traversal (`..`) returns 400 `ZIP_PATH_TRAVERSAL`
- Given a batch with mixed valid/duplicate/failed files, then `summary.total`, `summary.imported`, `summary.duplicates`, `summary.failed` all accurate
- Given `POST /api/v1/submissions/import-duplicate` with `batchId` + `filename`, then the duplicate is force-imported; `status: "IMPORTED"`

**Auto-Grading — Blockly (Rhino):**
- Given a Blockly submission with `outputMatch` rule and correct output, then that aspect passes; score proportional to weight
- Given `requiredBlocks` rule enabled and a required block absent from workspace XML, then that aspect fails
- Given code running longer than 3 seconds in Rhino, then output match fails with `error: "TIME_LIMIT_EXCEEDED"`; block-check aspects still evaluate
- Given all aspects pass, then `autoScore = 100.0`; partial passes produce proportional scores

**Auto-Grading — Python (Sandbox):**
- Given a Python submission, when graded, then all test cases (visible + hidden) sent to `POST sandbox:5000/execute`
- Given sandbox returns results, then `autoScore = (passedCount / totalCount) * 100`; each test case result stored in `submissions.auto_grade_details` JSON
- Given sandbox unavailable, then submission saved with `autoScore = null`, `error: "SANDBOX_UNAVAILABLE"`; import does not fail wholesale
- Given a test case times out, then that case: `passed: false`, `error: "TIME_LIMIT_EXCEEDED"`; remaining cases continue

**Review & Manual Grade:**
- Given `GET /api/v1/submissions/{id}`, then returns all grading details including per-test-case breakdown (Python) or per-rule breakdown (Blockly); `versionMismatch: true` if student's version ≠ current version
- Given `PUT /api/v1/submissions/{id}/grade` with `tutorScore` and `tutorComment`, when saved, then tutor score takes priority over auto score in all displays
- Given `versionMismatch = true`, when submission detail opens, then banner: "This submission was answered against version N. The exercise has since been updated to version M."

**CSV Export:**
- Given `GET /api/v1/submissions/export-csv`, when called, then CSV streamed with columns: Student Name, Exercise Title, Exercise Type, Auto Score, Tutor Score, Tutor Comment, Submitted At
- Given `?exerciseId=5` filter, then only submissions for exercise 5 included
- Given tutor score is null, then Tutor Score cell is empty (not "null")

### Tech Constraints

- Rhino 1.7.x; Java class access disabled via `Context.setClassShaper()`; instruction count limit ≈ 3-second equivalent (empirically determined, configurable); block analysis on workspace XML via DOM parsing (no Blockly library needed server-side); score formula: `(passedAspects / totalAspects) * 100`, rounded to 2 decimal places
- Python grading: Spring `RestTemplate` or `WebClient` (blocking acceptable); sandbox timeout: `timeLimitSeconds + 2s` network buffer; `auto_grade_details` stored as JSON column in `submissions` table
- Max ZIP decompressed size: 100 MB (configurable); exceeded → 400 `ZIP_TOO_LARGE`; max files in ZIP: 500
- JSON schema validation: required fields `exerciseId`, `exerciseType`, `studentName`, `answer`, `exportedAt`
- Processing is synchronous per file (acceptable for single-university batch sizes)
- `batchId` (UUID) generated per import request; stored in server-side session / short-lived cache (5-minute TTL) for use by force-import endpoint
- `versionMismatch` computed at query time: compare `submissions.exercise_version_id` with `exercises.current_version_id`
- CSV: Apache Commons CSV 1.11.x; `Content-Type: text/csv; charset=UTF-8`; `Content-Disposition: attachment; filename="grades_YYYY-MM-DD.csv"`; stream directly to `HttpServletResponse`; do not buffer
- Import UI: drop zone accepts `.json` and `.zip` only (MIME type + extension); no page reload on completion; summary bar: "X imported, Y duplicates, Z failed"
- Submission detail UI: read-only Blockly workspace (disable all editing interactions); read-only Monaco (`readOnly: true`); tutor score: number input 0–100, decimal allowed; comment: textarea max 500 chars
- CSV export via `<a href="/api/v1/submissions/export-csv?exerciseId=X" download>` — auth handled via cookie-based approach (cannot set headers on `<a download>`)
- Rate limit on import: 5 req/min per user (Bucket4j)

### Out of Scope

- Student online submission (turn-in) — stays in OLE
- Plagiarism detection
- Bulk re-grading after exercise edit
- LTI integration
